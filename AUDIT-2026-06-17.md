## 1. 요약 (TL;DR)

### 문서 (DOCS) — 다수 P0, 런타임/import 즉시 실패
- **`chart.getData()`가 15개 문서·42회 등장하지만 존재하지 않는 API.** 근본 원인은 `api/chart-handle.mdx:20`이 `getData: () => Dataset`를 `ChartHandle` 인터페이스로 *정의*한 것 — 나머지 페이지가 전부 복사. 실제 핸들엔 없음(`chart-handle.ts:25-90`), 데이터는 `getState().dataset`로만 노출(`create-chart.ts:164-170`).
- **published 서브패스/UMD import가 해소 불가.** `@xelstack/chart-core/utils`·`/effects`는 exports에 없고(`package.json:9-35`), `installation.mdx:49`가 광고하는 `dist/index.umd.js`는 **빌드 산출물에 없음**(rollup이 esm/cjs index만 emit, `rollup.config.js:34-60`). 심지어 advertised 서브패스(`./charts/line` 등)도 JS가 emit 안 됨(`preserveModules` 부재).
- **미존재 API/타입 광범위 문서화.** `type Chart`(실제 `ChartHandle`), `LineChartConfig`/`BarChartConfig`, `virtualizeDataset`/`getDeviceInfo`/`freeze`/`clone`/`getMinMax` 등.
- **`ChartConfig` 스타일 필드 허위.** `lineWidth`/`pointRadius`/`barWidth`/`innerRadius`/`showLabels`/`orientation` 등은 `ChartConfig`에 없고 렌더러로도 안 감(`types/index.ts:119-142`, `canvas-render.ts:97-136`). donut/수평막대/포인트크기 예제 전부 무효과.
- **시그니처 드리프트.** 순수 함수 대부분이 `Dataset`이 아닌 `DataPoint[]`를 받는데 문서는 `Dataset` 전달(`data-transforms.ts`, `sampling.ts`, `viewport-calculations.ts`).

### 함수형 생태계 (FP) — 기반 견고, toolkit 얇음
- `pipe`/`compose`(arity 1-10, 무할당 루프), `deepFreeze`(순환 안전), 순수 transform/state/viewport 레이어는 실제로 잘 구현됨.
- **`Result`/`Either`/`Option` ADT 0건이 최대 철학적 모순.** 검증은 throw 기반 assertion(`validation.ts:14,50,83,118`)뿐 → FP 소비자가 `pipe` 안에서 try/catch 없이 합성 불가.
- **`Immutable<T>`/`ImmutableArray<T>`가 정의됐으나 public entry에서 re-export 안 됨**(`types/fp.ts:53-60`) → 소비자 도달 불가.
- **`getState()`가 내부 live 객체를 그대로 반환**(`create-chart.ts:164-170`, `data-manager.ts:66`) → 소비자 mutation이 내부 상태 오염, 불변성 보증 위반.
- `curry`는 binary 전용(`curry.ts:14`). (※ 문서의 3-arg 광고 주장은 근거 약함 — `functional-programming.mdx`에 curry 예제 없음. 구현/문서 정합만 점검 권장.)

### 성능 (10MB/s @ 60fps) — 구조적으로 불가능 (현재 아키텍처)
- 첫 poll부터 메인스레드 동기 작업 **~30-80ms vs 16.67ms 예산 = 2-5배 초과**, 시간이 지날수록 단조 증가.
- per-frame 비용이 **화면 픽셀이 아니라 누적 포인트 수**에 비례: 버퍼 기본 무제한(`incremental-renderer.ts:64`) + 매 프레임 전체 redraw(`:100`).
- **"off-thread incremental"은 이름뿐.** Worker/OffscreenCanvas API/transferControlToOffscreen/postMessage src 0건. `offscreen-canvas.ts:42`는 `document.createElement('canvas')`(in-DOM). `renderIncrementalToCanvas`·`performance-monitor.ts` 모두 export되나 호출처 0 (dead code).
- typed array/LTTB/min-max 샘플링 src 0건. 모든 데이터 AoS `DataPoint[]`, poll마다 3회 full copy(`data-buffer.ts:56,83`, `incremental-renderer.ts:93`).

### 🎯 단 하나의 가장 중요한 액션
> **typed-array SoA 링 버퍼 + 기본 윈도우 상한 도입**(`data-buffer.ts`, `incremental-renderer.ts:64`). 성능 목표의 키스톤이며 불변성/GC/메모리 문제를 동시 해소. 나머지 P0(증분 렌더, 다운샘플)는 모두 이 위에서 쉬워짐.

## 4. 성능: 10MB/s @ 60fps 달성 가능성

### 가정
- `DataPoint`는 이종 V8 객체(`types/index.ts:23-34`). wire JSON ≈25-40 B/pt → **~250k pt/poll**, 힙 ≈80-120 B/pt → live ~25-40MB. 작업 기준 **N≈250k**(±2×, verdict는 정밀도 무관).
- 버퍼 기본 무제한(`incremental-renderer.ts:64`) → 지속 시 N = 250k × 경과초.

### 프레임 예산 (16.67ms, N=250k) — ms는 모두 **추정치**

| 스테이지 | 이상 | 현재(추정) | 초과 | 근거 |
|---|---|---|---|---|
| Ingest/parse | ~3ms(off-thread) | 메인 `JSON.parse` 15-40ms(소비자) | ~5-13× | lib에 parse 경로 0 |
| 버퍼 복사 | ~0.5ms | 3회 full O(N) spread 8-20ms | ~16-40× | `data-buffer.ts:56,83`; `incremental-renderer.ts:93,167` |
| 델타 diff | ~1ms | append 1×O(N); replace 3×O(N) 15-45ms | ~5-45× | `delta-calculator.ts:84-152` |
| 샘플 | ~2ms | 렌더 경로에 없음(dead) | n/a | `sampling.ts` export, 내부 호출 0 |
| Cull | ~0.5ms | O(N) 선형 스캔 매 프레임, 무제한 증가 | YES | `virtualization.ts:63-79` |
| Draw | ~6ms | 단일 stroke, 1000 cap 0.1-0.5ms | no(양호) | `line.ts:76-126` |
| GC | ~1ms | poll마다 25-40MB 폐기 → major GC pause | YES | `incremental-renderer.ts:93` |
| **TOTAL** | **16.67ms** | **~30-80ms, 매초 증가** | **2-5× → 무제한** | |

### 정직한 verdict
> **불가능 — 상수배가 아니라 구조적으로.** 지배 병목은 AoS `DataPoint[]` + full-copy + full-redraw-of-everything. 최악 단일 오펜더는 무제한 버퍼 위 per-frame O(N_total)(`incremental-renderer.ts:100`→`virtualization.ts:63`). 화면 픽셀 한정(~1ms) 대비 10× 이상 벗어남.

### 필요 아키텍처
**A. typed-array SoA 링 버퍼**(`xs/ys: Float64Array`) — append `array.set`, trim tail index, O(k) 무할당. **키스톤.**
**B. 기본 윈도우 상한** — `maxPoints` 실제 기본값(픽셀폭 기반), oldest drop. per-frame을 세션 길이와 독립화.
**C. 진짜 Worker + OffscreenCanvas** — `transferControlToOffscreen()`, parse+diff+downsample off-thread, ArrayBuffer zero-copy.
**D. 화면해상도 다운샘플(min/max-per-pixel 또는 LTTB)** — poll당 1회, `line.ts:61` 연결. 현재 blind stride는 peak 소실(aliasing).
**E. sorted-x binary-search cull** — `virtualization.ts:63` 선형→O(log N+visible), `[lo,hi]` 프레임 캐시.
**F. 실제 증분 렌더 + back-pressure** — `renderIncrementalToCanvas`를 `onFrameRender`(`:100`) 연결; `createRenderQueue` maxSize·`pendingPoints` 한정(`frame-scheduler.ts:159`).

### 액션 플랜
**P0**: ① typed-array 링 버퍼+상한 ② 매 프레임 full redraw 중단→증분 draw ③ 화면해상도 다운샘플 연결.
**P1**: ④ 진짜 Worker+OffscreenCanvas ⑤ binary-search cull ⑥ 델타 O(1) append + caller copy 제거.
**P2**: ⑦ back-pressure ⑧ device 메모이즈 ⑨ dirty-rect ⑩ dead sampling 정리.
