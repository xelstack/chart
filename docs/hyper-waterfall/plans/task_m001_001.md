# task_m001_001 수행계획서

## Goal

도메인 문서만 남은 repository에서 TypeScript chart library의 첫 프로젝트 골격을 세운다.

이 task의 목표는 라이브러리 기능 구현이 아니라, 사용자가 이후 코드를 직접 이해하고 칠 수 있는 최소 개발 환경과 디렉터리 경계를 결정하는 것이다.

최종 산출물은 다음 두 가지다.

- 이 수행계획서: `docs/hyper-waterfall/plans/task_m001_001.md`
- 승인 후 작성할 구현계획서: `docs/hyper-waterfall/plans/task_m001_001_impl.md`

## Scope

이번 task에서 결정한다.

- 패키지 매니저
- pnpm workspace 구조
- TypeScript 설정 방향
- 테스트 도구
- 빌드 도구
- 첫 디렉터리 구조
- 사용자가 처음 직접 칠 코드 단위
- 검증 명령의 최소 루프

이번 task에서 구현계획서 승인 후 만들 수 있는 파일 범위는 다음으로 제한한다.

- `package.json`
- `pnpm-workspace.yaml`
- `pnpm-lock.yaml`
- root TypeScript/lint/test 설정 파일
- `packages/chart/package.json`
- `packages/chart/tsconfig.json`
- lint/format/test/build 설정 파일
- `packages/chart/src/`의 최소 entry와 domain type 파일
- `packages/chart/src/` 안의 colocated type/contract test 파일

## Non-Goals

이번 task에서는 다음을 하지 않는다.

- `createChart` 구현
- Canvas renderer 구현
- realtime append buffer 구현
- downsampling 구현
- RAF scheduler 구현
- public API 문서 완성
- benchmark harness 구현
- Chart.js 호환성 계층 구현
- 도메인 주도 설계를 위한 함수형 프로그래밍 패턴 도입
- `Result`/ADT/railway-oriented programming 같은 도메인 workflow 구현 패턴 결정
- `fp-ts`, `Effect` 같은 함수형 라이브러리 도입

## Domain Sources

이번 계획은 다음 문서를 기준으로 한다.

- `docs/domain/01-event-storming.md`
- `docs/domain/02-context-map.md`
- `docs/domain/03-domain-model.md`
- `docs/domain/04-invariants.md`
- `docs/domain/05-workflows.md`
- `docs/domain/06-api-contract.md`
- `docs/hyper-waterfall/orders/2026-06-19-realtime-line-area-domain.md`
- `docs/hyper-waterfall/orders/20260620.md`

중요한 도메인 제약:

- v1은 realtime `line`과 `area`에 집중한다.
- public API는 Chart.js-like이지만 Chart.js-compatible은 아니다.
- `append`가 realtime canonical path다.
- typed-array append가 hard performance path다.
- 내부 storage는 series-local columnar buffer다.
- render는 `requestAnimationFrame` scheduler가 담당한다.
- public `flush()`는 제공하지 않는다.
- downsampling은 내부 자동 정책으로 시작하며 기본 후보는 M4다.

## Tooling Decisions

### Package Manager

`pnpm`을 사용한다.

Reason:

- 사용자가 `pnpm`을 package manager로 선택했다.
- dependency 설치와 lockfile 관리를 `pnpm-lock.yaml` 하나로 고정한다.
- repository 시작부터 `pnpm-workspace.yaml`을 두어 package boundary를 명확히 한다.
- root는 orchestration만 담당하고 실제 library code는 workspace package 안에 둔다.

### Workspace Layout

pnpm workspace를 처음부터 사용한다.

초기 workspace package:

```text
packages/chart
```

root package는 private orchestration package로 둔다.

Responsibilities:

- 공통 dev dependency 관리
- workspace script 실행
- root-level quality gate 제공

`packages/chart`는 실제 publish 대상 library package로 둔다.

Initial package name:

```text
@xelstack/chart
```

Responsibilities:

- public API export
- domain type
- runtime implementation
- package-local tests
- package-local build output

초기에는 package를 하나만 만든다. 다만 이후 다음 package를 자연스럽게 추가할 수 있게 한다.

```text
packages/bench
packages/examples
packages/devtools
```

### TypeScript

TypeScript는 최신 stable release를 기준으로 한다.

Current decision:

```text
typescript@6.0.3
```

이 버전은 2026-06-22 기준 npm registry의 `typescript/latest`에서 확인한 최신 버전이다.

구현 시 package manifest에는 재현성을 위해 `latest`가 아니라 확인된 버전 `6.0.3`을 명시한다.

초기 설정 방향:

- strict mode 사용
- browser library를 전제로 DOM type 포함
- root에는 shared config를 둔다.
- source는 `packages/chart/src/`
- unit/domain test는 source 옆에 colocated test로 둔다.
- library output은 `packages/chart/dist/`
- public type export를 처음부터 의식한다.

### Test Tool

초기 unit test는 `vitest`를 사용한다.

Reason:

- TypeScript 테스트 작성이 단순하다.
- Node 환경에서 도메인 타입, normalization, buffer logic을 빠르게 검증할 수 있다.
- workspace package 단위로 test를 실행하기 쉽다.
- Canvas와 real-browser 검증은 이후 별도 task에서 Playwright 기반으로 추가한다.

### Build Tool

초기 library build는 `tsup`을 후보로 둔다.

Reason:

- TypeScript library의 ESM/CJS/type declaration output을 단순하게 만들 수 있다.
- runtime 성능과 bundler 선택을 섞지 않을 수 있다.
- 복잡한 app bundler 설정 없이 package 형태를 빠르게 검증할 수 있다.

단, Stage 1에서는 `tsc --noEmit` typecheck 루프를 먼저 세우고, 실제 bundle output은 구현계획서에서 별도 단계로 나눈다.

### Lint

초기 lint는 `eslint`와 TypeScript ESLint를 사용하되, rule set은 최소로 시작한다.

Reason:

- `pnpm lint`를 명시적으로 유지할 수 있다.
- root script에서 workspace package lint를 실행한다.
- 성능 중심 코드에서 암묵적 `any`, unused branch, 잘못된 promise 사용 등을 초기에 차단할 수 있다.
- 세부 lint rule은 실제 생성되는 코드를 보면서 필요한 것만 추가한다.
- lint를 초기에 과하게 잡아 코드 이해와 타이핑 흐름을 방해하지 않는다.

### Formatter

Prettier를 사용한다.

Config file:

```text
prettier.config.js
```

Initial config:

```js
/** @type {import("prettier").Config} */
const config = {
  semi: true,
  tabWidth: 2,
  useTabs: false,
};

export default config;
```

Reason:

- formatting 판단을 ESLint rule과 분리한다.
- 세미콜론을 항상 사용한다.
- 들여쓰기는 2칸으로 고정한다.
- 사용자가 직접 치는 코드와 Codex가 리뷰하는 diff의 모양을 안정적으로 유지한다.
- root `package.json`이 `"type": "module"`이므로 `prettier.config.js`는 ESM config로 동작한다.
- CommonJS config가 필요해질 때만 `prettier.config.cjs`로 바꾼다.

## First Human-Typed Code Unit

첫 코드 단위는 `packages/chart/src/domain/public-types.ts`로 한다.

이유:

- 도메인 문서의 public contract를 TypeScript 타입으로 옮기는 가장 작은 단위다.
- Canvas, scheduler, renderer, buffer 구현 없이도 이해하고 검증할 수 있다.
- 이후 `createChart`, `append`, storage 구현이 이 타입을 기준으로 이어진다.

## Stages

### Stage 1: Toolchain Skeleton

목표:

- pnpm 기반 TypeScript project를 초기화한다.
- pnpm workspace를 초기화한다.
- typecheck, test, lint 명령을 만든다.

예상 파일:

- `package.json`
- `pnpm-workspace.yaml`
- `pnpm-lock.yaml`
- `tsconfig.base.json`
- `packages/chart/package.json`
- `packages/chart/tsconfig.json`
- eslint 설정
- prettier 설정
- vitest 설정

Gate:

- `pnpm typecheck`
- `pnpm test`
- `pnpm lint`
- `pnpm format:check`

Root scripts should delegate to workspace packages.

```text
pnpm -r typecheck
pnpm -r test
pnpm -r lint
pnpm -r format:check
```

### Stage 2: Source Boundary Skeleton

목표:

- `packages/chart/src/`의 첫 source와 colocated test 구조를 만든다.
- public type entry를 만든다.

예상 파일:

- `packages/chart/src/index.ts`
- `packages/chart/src/domain/public-types.ts`
- `packages/chart/src/domain/public-types.test.ts`

Gate:

- public type이 `packages/chart/src/index.ts`에서 export된다.
- 테스트가 타입/런타임 최소 계약을 확인한다.
- 아직 `createChart` 구현은 없다.

Test placement rule:

- unit/domain tests are colocated next to source files.
- integration/API contract tests may use a separate `tests/` directory in a future task.
- browser/canvas/performance tests stay out of this package skeleton and will be planned separately.

### Stage 3: Stage Report

목표:

- 실제 생성된 파일과 계획 차이를 기록한다.

예상 파일:

- `docs/hyper-waterfall/working/task_m001_001_stage1.md`

Gate:

- 사용자가 stage report를 확인하고 다음 task로 넘어갈지 판단한다.

## Risks

### Risk: 문서가 구현보다 커질 수 있음

대응:

- 수행계획서는 도구와 단계까지만 정한다.
- 구현계획서에서는 사용자가 직접 칠 수 있는 작은 코드 블록으로 쪼갠다.

### Risk: 초기에 도구 설정이 과해질 수 있음

대응:

- browser test, benchmark, renderer 검증은 이후 task로 분리한다.
- 이번 task는 unit test와 typecheck 루프까지만 만든다.
- workspace package도 처음에는 `packages/chart` 하나만 만든다.
- lint rule은 최소로 시작하고, 실제 코드에서 반복되는 문제가 보일 때 추가한다.
- formatter는 Prettier로 분리해 lint rule과 충돌하지 않게 한다.

### Risk: 최신 TypeScript와 주변 도구 호환성이 어긋날 수 있음

대응:

- TypeScript는 확인된 최신 버전 `6.0.3`으로 pin한다.
- eslint, TypeScript ESLint, tsup, vitest는 구현계획서에서 호환 가능한 최신 조합으로 고른다.
- 호환성 문제가 나오면 조용히 downgrade하지 않고 troubleshooting 문서에 원인과 선택지를 기록한다.

### Risk: public type이 내부 최적화를 막을 수 있음

대응:

- `public-types.ts`와 `internal-types.ts`를 분리한다.
- Chart.js-like config는 normalization 경계 밖으로 직접 흘러가지 않게 한다.
- 함수형 도메인 모델링 기법은 이번 task에서 채택하지 않고, 다음 domain type task에서 가벼운 `functional core, imperative shell` 원칙으로 재검토한다.

### Risk: 함수형 도메인 모델링을 너무 일찍 도입할 수 있음

대응:

- 이번 task는 project skeleton과 public type boundary에 집중한다.
- hot path에는 allocation과 abstraction cost가 직접 영향을 주므로, DDD/FP 패턴은 성능 측정 가능 지점이 생긴 뒤 도입 여부를 판단한다.
- 다음 task에서 `Command`, `Event`, validation result, branded type을 어디까지 쓸지 별도로 계획한다.

### Risk: workspace가 단일 package 프로젝트보다 복잡할 수 있음

대응:

- root는 orchestration만 담당한다.
- 실제 코드는 `packages/chart`에만 둔다.
- 새 package는 benchmark, example, devtool처럼 경계가 생길 때만 추가한다.
