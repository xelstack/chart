# task_m001_002 수행계획서

## 목표

`task_m001_002`의 목표는 public type을 한 단계 정리하고, public config가 internal hot path로 직접 흘러가지 않도록 config normalization boundary를 정의하는 것이다.

이 task는 `createChart` 구현으로 들어가기 전의 경계 설계 작업이다.

핵심 질문은 다음이다.

- 사용자는 Chart.js처럼 쉬운 config를 넘긴다.
- 내부 엔진은 series-local columnar buffer, ordered append, RAF scheduler, downsampling에 맞는 normalized model을 원한다.
- 그러면 public config를 어디서 끊고 어떤 internal model로 바꿔야 하는가?

## 범위

이번 task에서 결정한다.

- `ChartOptions`를 지금 public type에 넣을지 보류할지
- initial dataset data와 realtime append input을 같은 타입으로 계속 둘지, 이름을 분리할지
- `DatasetConfig.id`가 없는 경우 normalization boundary에서 어떤 internal `SeriesId`를 만들지
- public type과 internal type의 export 경계
- config normalization의 최소 책임
- validation result의 최소 표현
- public package entry contract test를 이번 task에 포함할지

이번 task에서 구현계획서 승인 후 만들 수 있는 코드 범위는 다음으로 제한한다.

- `packages/chart/src/domain/public-types.ts`의 public type refinement
- `packages/chart/src/domain/internal-types.ts`의 첫 internal normalized config type
- `packages/chart/src/config/` 아래의 config normalization boundary
- config normalization unit test
- `packages/chart/src/index.ts`의 public export 조정

## 결정 방향

### `ChartOptions`

이번 task에서는 `ChartOptions`를 public type에 추가하지 않는다.

이유:

- scale, style, interaction option의 의미가 아직 구현과 연결되지 않았다.
- 빈 `ChartOptions` interface는 사용자에게 실제 지원하지 않는 확장 가능성을 암시할 수 있다.
- v1의 첫 public contract는 `type`, `realtime`, `data.datasets`에 집중하는 편이 더 선명하다.

나중에 renderer, scale, interaction 경계가 실제 구현 의미와 함께 생기면 필요한 option만 추가한다.

### Initial data와 append input

`AppendInput`은 `chart.append(seriesId, input)`의 realtime canonical path를 표현한다.

`DatasetConfig.data`는 initial dataset input이다.

두 값의 물리적 shape가 같더라도 의미는 다르므로, 이번 task에서는 이름을 분리한다.

후보:

```ts
export type InitialDataInput = AppendInput;
```

이렇게 하면 runtime 변환 비용 없이 의미만 분리할 수 있다. 이후 initial data에서만 허용하거나 append에서만 허용하는 shape가 생기면 자연스럽게 타입을 갈라낼 수 있다.

### Dataset id와 SeriesId

`DatasetConfig.id`는 optional로 유지한다.

`id`가 없는 dataset은 config normalization boundary에서 deterministic internal `SeriesId`를 생성한다.

예상 정책:

```text
datasets[0] without id -> "series-0"
datasets[1] without id -> "series-1"
```

Realtime append를 사용할 사용자는 명시적 `id`를 쓰는 것을 문서와 예제에서 권장한다.

`SeriesId`는 이번 task에서 단순 string으로 둔다.

```ts
export type SeriesId = string;
```

branded type은 id 혼동이 실제 문제가 되거나 internal model이 커진 뒤 다시 검토한다.

### Internal normalized config

config normalization boundary의 첫 internal model은 다음 정도로 제한한다.

- `SeriesId`
- `SeriesDescriptor`
- `NormalizedRealtimeOptions`
- `NormalizedDatasetConfig`
- `NormalizedChartConfig`
- `ConfigValidationIssue`
- `ConfigValidationResult`

이 model은 public export가 아니다.

`NormalizedChartConfig`, `SeriesDescriptor`, `ConfigResult` 같은 internal type은 root package entry에서 export하지 않는다. 내부 test는 필요한 경우 internal module을 직접 import할 수 있지만, 사용자-facing API에는 노출하지 않는다.

### Public package entry contract test

이번 task에서는 public package entry contract test를 보류한다.

조사 결과:

- TanStack Query와 Effect는 타입 테스트를 적극적으로 사용한다.
- 하지만 확인한 범위에서 단순히 `index.ts` re-export 여부만 검증하는 테스트가 핵심은 아니다.
- 두 프로젝트 모두 공개 API의 타입 추론, generic 제약, narrowing, error expectation 같은 의미 있는 타입 동작을 테스트한다.

결정:

- `packages/chart/src/index.test.ts`는 이번 task에서 만들지 않는다.
- `InitialDataInput` export 여부는 `pnpm typecheck`와 기존 domain type test로 충분히 확인한다.
- public runtime API, package exports, build output boundary가 생긴 뒤 package-level contract test를 추가한다.
- build/publish boundary task에서는 `publint`, `attw` 같은 package entry 검증 도구도 함께 검토한다.

### Validation result

이번 task에서는 외부 library 없이 단순 Result 계열 discriminated union을 사용한다.

예상 형태:

```ts
type ConfigResult<T> =
  | { ok: true; value: T }
  | { ok: false; issues: ConfigValidationIssue[] };
```

이것은 full DDD/FP 또는 Effect runtime 도입이 아니라 config boundary에서 실패를 안전하게 표현하기 위한 최소 장치다.

장기적으로는 Effect식 사고방식을 참고할 수 있다.

- 실패를 값으로 표현한다.
- boundary에서 validation을 명확히 한다.
- public API는 단순하게 유지하고 internal workflow는 합성 가능하게 만든다.

하지만 이번 task에서는 Effect dependency를 도입하지 않고, Effect 코드를 복사하지도 않는다. `ok`, `fail` helper는 필요할 때 최소 범위로만 추가한다.

### Config normalization 최소 책임

이번 task에서 normalization이 책임질 수 있는 최소 범위:

- chart type 검증
- realtime option 기본값 적용
- dataset을 stable internal series descriptor로 변환
- dataset `id`가 없을 때 deterministic internal id 생성
- public dataset data를 internal normalized dataset input으로 넘길 boundary 확보

이번 task에서 하지 않을 normalization:

- point x/y encode
- `Date` to epoch milliseconds 변환
- `null` to `NaN` gap encoding
- ordered invariant 검증
- buffer allocation

이 작업들은 input normalization 또는 storage task로 분리한다.

## 위험

### 위험: `ChartOptions`를 너무 일찍 넓힐 수 있음

대응:

- 구현 semantics가 없는 option은 public type에 넣지 않는다.
- future option은 문서에만 남기고 code contract로 고정하지 않는다.

### 위험: public type이 internal hot path를 오염시킬 수 있음

대응:

- public config는 config normalization boundary에서 끊는다.
- internal normalized type은 public export에 포함하지 않는다.

### 위험: validation result가 과한 추상화가 될 수 있음

대응:

- library 없이 단순 discriminated union만 사용한다.
- result type은 config normalization boundary 안에서만 사용한다.

### 위험: task 범위가 `createChart` 구현까지 커질 수 있음

대응:

- 이번 task는 normalize function과 type/test 경계까지만 다룬다.
- instance lifecycle은 다음 task로 분리한다.
