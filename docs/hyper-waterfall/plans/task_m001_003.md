# task_m001_003 수행계획서

## 목표

`task_m001_003`의 목표는 import policy와 module boundary tooling을 정리하는 것이다.

`task_m001_002`에서 package-local alias `@/* -> ./src/*`를 추가했고, 내부 구현은 이미 `@/domain/*`, `@/config/*` 형태를 사용하기 시작했다.

이번 task에서는 이 alias가 우연히 들어간 편의 기능이 아니라, public/internal boundary를 지키는 규칙으로 동작하도록 정리한다.

핵심 질문은 다음이다.

- 내부 source import에는 언제 `@/*` alias를 쓰는가?
- public package entry인 `src/index.ts`는 어떤 import/export 규칙을 가져야 하는가?
- package 내부에서 `@xelstack/chart` self-import를 허용할 것인가?
- test는 public entry 기준 import와 internal module import를 어떻게 구분할 것인가?
- import 정렬은 수동 규칙으로 둘 것인가, tooling으로 자동화할 것인가?
- TypeScript, Vitest, package build는 같은 import 정책을 이해하는가?

## 현재 코드 상태

현재 import 관련 상태:

- `packages/chart/tsconfig.json`에 `@/* -> ./src/*` paths alias가 있다.
- `packages/chart/vitest.config.ts`에 `@` alias가 있다.
- `packages/chart/src/config/normalize-config.ts`는 `@/domain/*` alias를 사용한다.
- `packages/chart/src/config/normalize-config.test.ts`는 `@/config/*`, `@/domain/*` alias를 사용한다.
- `packages/chart/src/index.ts`는 root public entry이고 relative export를 사용한다.
- import 정렬 자동화는 아직 없지만, 이번 task에서 도입하기로 결정했다.

- public entry에서 alias를 써도 되는지 기준이 없다.

- `tsup --dts` build가 TypeScript 6에서 실패했으므로 `tsdown` 전환을 구현계획서에 반영해야 한다.
- package build가 alias를 실제 publish output에서 어떻게 처리하는지 task 단위로 검증해야 한다.

## 결정 방향

### Alias 사용 규칙

초기 후보:

- 내부 source module은 `@/*` alias를 사용할 수 있다.
- 같은 directory 안의 짧은 relative import는 허용할지 결정한다.
- root public entry인 `src/index.ts`는 public boundary로 보고, internal alias 사용을 제한할지 결정한다.
- package 내부 source에서 `@xelstack/chart` self-import는 금지 후보로 둔다.

이유:

- `@/*` alias는 내부 source layout을 읽기 쉽게 만들기 위한 장치다.
- `@xelstack/chart`는 사용자-facing package name이므로 package 내부 구현에서 쓰면 public API와 internal module 경계가 섞일 수 있다.

### Test import 경계

초기 후보:

- internal behavior test는 `@/config/normalize-config`처럼 internal module을 직접 import할 수 있다.
- public API contract test는 나중에 runtime public API가 생긴 뒤 package entry 기준으로 작성한다.
- 단순 re-export smoke test는 현재처럼 보류한다.

### Import 정렬

초기 후보:

- `@ianvs/prettier-plugin-sort-imports`를 도입한다.
- import group은 `side-effect -> external package -> @/* internal alias -> relative` 순서로 둔다.
- type import는 value import와 같은 source group 안에 둔다.
- side-effect import는 최상단에 두고 작성 순서를 보존한다.

주의:

- import sort plugin은 formatting tool이지만 dependency와 lockfile을 바꾸는 tooling 변경이다.
- plugin 설치는 구현계획서 승인 뒤 진행한다.

### Build/tooling 검증

이번 task에서 확인해야 할 도구:

- TypeScript `paths`
- Vitest `resolve.alias`
- tsdown build
- Prettier
- ESLint

특히 `tsdown` build가 alias/import policy와 충돌하지 않는지 확인한다.
